# API 与对外契约说明

本文档不是单纯的接口清单，而是说明：**这个 FastAPI 接口层在 DDD / 六边形架构里到底扮演什么角色**。

如果只看路由，这个项目很像一个任务 CRUD；但从教学视角，它更重要的是演示：

- HTTP 层如何作为**入站适配器（Inbound Adapter）**
- 请求 / 响应模型如何作为**外部契约模型（External Contract Model）**
- 外部模型如何在边界处翻译为应用层命令与查询
- 领域模型如何保持**不直接暴露给外部**
- 错误码如何表达“输入错误 / 业务错误 / 资源状态错误”的不同语义

---

## 1. 这一层在 DDD 里是什么

### 1.1 HTTP 层不是业务层

`src/task_management/interfaces/http/` 目录中的 FastAPI 路由，不负责承载核心业务规则。

它主要做 4 件事：

1. 接收 HTTP 请求
2. 校验外部输入格式
3. 把外部请求模型翻译成应用层命令 / 查询
4. 把应用层返回结果翻译成稳定的 HTTP 响应契约

也就是说，HTTP 层更像一个**翻译器 + 协议适配器**，而不是“把业务全写在路由里”。

### 1.2 为什么这可以看作 ACL 的最小版本

在 DDD 语境里，ACL（Anti-Corruption Layer，防腐层）常用于隔离外部系统模型与内部领域模型。

在这个教学仓库里，HTTP 接口层虽然不是典型的“系统对系统集成 ACL”，但它承担了同一种**防污染职责**：

- 外部世界讲的是 JSON、状态码、字段可空性、查询参数
- 内部领域讲的是 `Task`、`TaskTitle`、`TaskDescription`、领域异常与状态迁移

因此可以把这里的接口层理解为：

> **面向 HTTP 调用方的轻量 ACL / 契约边界层**。

它防止下面这些污染直接进入领域层：

- Web 框架的校验错误结构
- HTTP 特有的状态码语义
- “空字符串但业务上其实等于没填” 这类输入噪音
- 对外响应为了易用而设计的扁平 JSON 结构

## 2. 请求到领域的翻译链路

一次 `POST /api/v1/tasks` 的调用，大致会经历下面这条路径：

```text
HTTP JSON
  -> CreateTaskRequest
  -> CreateTaskCommand
  -> CreateTaskUseCase
  -> Task.create(...)
  -> TaskView
  -> TaskResponse
  -> HTTP JSON
```

这条链路刻意把“外部模型”和“内部模型”拆开。

## 2.1 外部请求模型：为契约而生

例如 `CreateTaskRequest`：

- 负责 HTTP 入参校验
- 负责把全空白字符串归一化
- 负责表达“这个字段在接口上是否必填、长度限制是多少”

它的设计目标不是代表领域对象，而是代表：

> “调用方发给 API 的数据长什么样。”

因此它属于**接口契约模型**，不是领域模型。

## 2.2 应用层命令：为用例而生

HTTP 层不会把 `CreateTaskRequest` 直接传进领域对象，而是先翻译为 `CreateTaskCommand`。

这样做的意义是：

- 应用层不依赖 FastAPI / Pydantic 的具体输入模型
- 同一个用例未来可以被 CLI、消息队列、定时任务复用
- 用例看到的是“创建任务”这件事，而不是“某个 HTTP body”

## 2.3 领域模型：为业务规则而生

真正的业务约束在领域模型中体现，例如：

- 标题不能为空
- 描述不能是纯空白字符串
- 指派人不能为空
- 已完成任务不能重复完成

领域层关心的是**业务语义是否成立**，而不是：

- 请求来自 REST 还是 gRPC
- 字段是从 JSON body 还是 query string 来的
- FastAPI 想返回什么错误格式

这就是边界的价值。

---

## 3. 为什么不能直接把领域模型暴露给接口层

教学型仓库最容易犯的错误之一，就是把领域对象直接拿去序列化返回。

这个项目刻意不这样做，因为这样会让几个边界混在一起：

### 3.1 外部响应稳定性会被内部重构绑架

如果直接返回领域对象，那么内部一旦：

- 调整值对象结构
- 修改枚举命名
- 新增内部字段
- 重构聚合表达方式

就可能意外影响 API 契约。

而 API 契约本应优先面向调用方稳定，而不是跟着内部重构一起抖动。

### 3.2 领域对象的“正确形状”不一定等于对外 JSON 的“好用形状”

例如领域层里：

- `TaskStatus` 是枚举
- `TaskDescription` 是值对象
- `AssigneeId` 是值对象

但对外响应中，调用方通常更希望拿到：

- 字符串状态值
- `description: string | null`
- `assignee_id: string | null`

因此项目使用 `TaskView` / `TaskResponse` 做一次显式投影：

- `TaskView`：应用层可读输出模型
- `TaskResponse`：HTTP 对外响应模型

这就是一种**有意识的外部模型翻译**。

## 4. 空值与归一化：边界层如何吸收输入噪音

### 4.1 description 为什么允许是 `null`

从调用方视角，任务描述经常是可选字段。

因此 API 契约允许：

- 不传 `description`
- 显式传 `null`
- 传非空字符串

但项目不希望“只包含空格的字符串”混入领域层，因为它既：

- 不是有意义的描述
- 也不是一个应该持久化保存的值

### 4.2 为什么空白描述在接口层被归一化为 `null`

`CreateTaskRequest` 会把仅包含空白字符的描述归一化为 `null`。

这样做的教学意义是：

- **外部契约允许宽松输入**，提升 API 易用性
- **内部领域保持严格语义**，不接纳无意义空字符串
- 归一化逻辑被放在边界层，而不是散落到业务流程各处

这就是一个很典型的“ACL 式输入整形”动作。

## 5. 错误码语义：不是为了凑状态码，而是为了表达边界

本项目统一错误结构：

```json
{
  "error": {
    "code": "REQUEST_VALIDATION_ERROR",
    "message": "请求参数校验失败",
    "details": []
  }
}
```

这里要区分两个层次：

- **HTTP 状态码**：给通用客户端、网关、监控系统看的协议语义
- **业务错误码 `error.code`**：给调用方程序和人类读者看的领域 / 契约语义

## 5.1 `422 REQUEST_VALIDATION_ERROR`

表示错误发生在**接口边界的输入校验阶段**。

典型场景：

- 缺少必填字段
- 字段类型不匹配
- 标题或指派人传了全空白字符串

这里的重点是：

> 请求甚至还没有以“合法命令”的身份进入应用层。

因此它代表的是**外部契约不成立**，不是领域规则失败。

## 5.2 `404 TASK_NOT_FOUND`

表示调用方引用了一个不存在的任务标识。

这是典型的资源寻址失败：

- URL 是合法的
- 请求格式也是合法的
- 但目标资源不存在

## 5.3 `409 TASK_ALREADY_COMPLETED`

表示调用方试图执行一个与当前资源状态冲突的动作。

例如：

- 第一次完成任务成功
- 第二次再完成同一任务，则返回 `409`

这类错误最适合表达：

> “请求格式没错，资源也存在，但它当前状态不允许这样做。”

## 5.4 `400 DOMAIN_ERROR`

这是一个教学型兜底：表示请求已通过接口层格式校验，但在进入领域规则后仍触发了业务异常。

它用于告诉读者：

- 输入合法 ≠ 业务合法
- 领域层仍然保有自己的最终校验权
- HTTP 层要把领域异常翻译成稳定对外契约

在更复杂项目里，`DOMAIN_ERROR` 往往还会继续细分为更明确的业务码；这个仓库先保留一个最小可理解版本。

---

## 6. 错误详情 `details` 的作用

`details` 主要服务于校验类错误，例如：

```json
{
  "error": {
    "code": "REQUEST_VALIDATION_ERROR",
    "message": "请求参数校验失败",
    "details": [
      {
        "field": "title",
        "message": "Value error, 标题不能为空",
        "type": "value_error"
      }
    ]
  }
}
```

教学上要强调的是：

- `message` 负责给人看整体语义
- `details` 负责给前端 / 调用方做字段级处理
- 这里返回的是**已经脱离 FastAPI 原始结构后的统一格式**

也就是说，项目没有把框架原生错误对象直接暴露给外部，而是做了一次**错误契约翻译**。

这同样属于 ACL 的职责范围。

## 7. 当前 API 清单

### 7.1 创建任务

`POST /api/v1/tasks`

请求体：

```json
{
  "title": "实现 ACL 文档",
  "description": "补充 API 与领域边界说明"
}
```

成功响应：`201 Created`

```json
{
  "data": {
    "id": "task_xxx",
    "title": "实现 ACL 文档",
    "description": "补充 API 与领域边界说明",
    "assignee_id": null,
    "status": "pending",
    "created_at": "2026-03-27T00:00:00Z",
    "updated_at": "2026-03-27T00:00:00Z",
    "completed_at": null
  }
}
```

契约要点：

- `title` 为必填
- `description` 为可选
- 空白 `description` 会被归一化为 `null`
- 返回的是对外响应模型，不是领域对象直接序列化结果

### 7.2 获取任务

`GET /api/v1/tasks/{task_id}`

成功响应：`200 OK`

失败：`404 TASK_NOT_FOUND`

### 7.3 列出任务

`GET /api/v1/tasks`

支持查询参数：

- `status`
- `assignee_id`

这里演示的是：HTTP query 参数先被翻译为 `ListTasksQuery`，再交给应用层处理。

### 7.4 指派任务

`POST /api/v1/tasks/{task_id}/assignments`

请求体：

```json
{
  "assignee_id": "user_001"
}
```

契约要点：

- `assignee_id` 必填
- 全空白字符串会在接口层被拒绝为 `422`
- 成功后返回任务最新视图

### 7.5 完成任务

`POST /api/v1/tasks/{task_id}/completion`

契约要点：

- 如果任务不存在，返回 `404`
- 如果任务已完成，返回 `409 TASK_ALREADY_COMPLETED`
- 体现的是“动作接口 + 状态迁移”，而不是只会改字段的 CRUD

### 7.6 健康检查

`GET /health`

用于运行时探活，不参与领域逻辑。

## 8. 从教学角度看，这套 API 为什么不像普通 CRUD

如果把它当 CRUD，会很容易只盯着“增删改查”本身；但这个仓库真正想强调的是：

### 8.1 接口表达的是用例，而不是数据库表操作

例如：

- `/assignments`
- `/completion`

它们表达的是**业务动作**，而不是“把某列改掉”。

这更贴近领域行为，也更有利于后续演进复杂规则。

### 8.2 状态变化由领域对象负责，不由路由直接拼装

路由不会写出类似：

- `task.status = "completed"`
- `task.completed_at = now`

这些状态迁移规则留在 `Task.complete()` 中，由领域对象统一维护。

### 8.3 对外契约与内部模型分离

项目刻意展示：

- request model ≠ command
- domain model ≠ response model
- framework error ≠ public error contract

这三次“不要直接穿透”的设计，就是教学价值本身。

---

## 9. 对前端 / 外部调用方的集成建议

如果你把这个仓库当成前后端协作样板，推荐这样理解：

- 以前端为代表的调用方，只依赖 HTTP 契约文档与错误码语义
- 不依赖后端内部领域对象命名细节
- 对 `422 / 404 / 409 / 400` 做分层处理
- 优先使用 `error.code` 做程序判断，而不是匹配自然语言 `message`

一个简单建议映射：

- `REQUEST_VALIDATION_ERROR`：前端做字段提示
- `TASK_NOT_FOUND`：提示资源不存在或已删除
- `TASK_ALREADY_COMPLETED`：提示当前状态不可重复操作，并刷新数据
- `DOMAIN_ERROR`：提示业务失败，并保留排查信息

## 10. 后续可以怎样继续演进

如果把这个教学样板再往前推进一小步，API 契约层还可以继续展示：

- 更细粒度的领域错误码映射
- OpenAPI 示例中的错误用例补全
- 仓储分页、游标与查询模型的契约设计
- 对外事件 / Webhook 模型与领域事件的映射关系
- 面向外部系统集成的真正 ACL 模块

但对当前仓库来说，现有设计已经足以传达一个核心观点：

> **优秀的 API 层不是业务逻辑的堆放处，而是外部契约与内部领域之间的翻译边界。**
