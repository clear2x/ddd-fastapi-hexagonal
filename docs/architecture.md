# 架构说明

这个项目用于演示一个**小而清晰**的 DDD + 六边形架构示例。

在当前版本里，仓库还额外强调两件教学上很关键、但常被省略的内容：

- `task_management` 是一个明确命名的**限界上下文（Bounded Context）**
- 外部系统与本上下文交互时，应通过**防腐层（ACL）**做语义翻译

## 分层说明

- **Domain**：业务模型、业务规则、不变量
- **Application**：用例编排、跨边界协作语义、依赖组装入口
- **Infrastructure**：持久化与运行时支撑
- **Interfaces**：HTTP API 与 ACL 适配器等边界适配层

可以把这四层理解成两次边界：

1. **外部世界 → 应用用例边界**：由 `interfaces` 负责翻译协议与契约
2. **应用用例 → 领域语义边界**：由 `application` 负责组织命令、查询与流程

## 依赖方向

外层依赖内层抽象，避免核心业务反向依赖框架与基础设施。

- interfaces -> application
- application -> domain
- infrastructure -> domain/application ports
- domain -> 不依赖外部框架

## 第二块增量：领域事件 + 读模型 + 教学版 CQRS

当前仓库在第二块 DDD 深化里，刻意补的是一条**最小但可读的 CQRS 教学主线**：

- 命令侧通过聚合执行业务动作
- 聚合产出领域事件
- 事件在进程内同步分发
- 查询侧把事件投影到 `task_read_models`
- `GET /tasks` / `GET /tasks/{id}` 从读模型返回结果

这里必须明确，它不是生产级消息架构，而是**同步、单进程、非事务 outbox 的教学版 CQRS**。

也就是说，当前示例**没有**：

- 事务 outbox
- 跨进程消息投递
- 消费重试 / 死信 / 补偿
- 独立读写存储扩缩容策略

保留这种克制，是为了让读者先看清“命令侧 vs 查询侧”的职责边界，而不是一开始就被工程化细节淹没。

## 为什么查询路径要切到读模型

这里让查询 use case 依赖 `TaskQueryService`，而不是直接读取 `TaskRepository`，是一个有意为之的教学信号。

它表达的是：

- **查询返回的不是聚合对象本身**，而是查询侧投影
- **查询契约面向展示与筛选**，因此允许围绕列表、状态、指派人等读取需求组织结构
- **命令侧的聚合边界不应该因为“查起来方便”就被 HTTP 查询直接穿透**

这也意味着当前实现有清楚的局限：

- 读模型依赖事件投影完成后才可见
- 如果投影缺失或损坏，写模型可能已成功，但查询侧仍读不到最新状态
- 当前示例通过显式失败暴露这种坏状态，但并不提供 outbox/重放/补偿来自动修复

因此，这里的核心收益不是“性能优化”，而是让读者看懂：

> 为什么查询路径不再直接查写库，以及读写分离一旦成立，就会带来独立的可见性边界与一致性约束。

## 限界上下文

当前仓库只有一个核心限界上下文：**任务管理（task_management）**。

它包含：
- 创建任务
- 获取任务
- 查询任务列表
- 指派任务
- 完成任务

这表示当前领域语言只在这个上下文内部成立。
未来如果引入其他上下文（例如用户中心、通知中心、项目协作），不应直接共享领域对象，而应通过边界协作完成集成。

## 防腐层（ACL）落点

当前仓库新增了两处 ACL 相关结构：

- `application/acl.py`：定义本上下文视角下的外部快照、翻译结果与翻译协议
- `interfaces/acl/`：放具体的边界翻译适配器示例

这样做是为了明确表达：

- 外部系统的“任务”不等于本上下文的 `Task`
- 外部状态、字段命名、标识规则不应直接穿透到领域层
- 翻译责任应位于边界，而不是散落在领域模型内部

## 接口层作为轻量 ACL 的位置

在更复杂的 DDD 系统里，ACL（Anti-Corruption Layer，防腐层）通常用于隔离外部系统模型与内部领域模型。

这个仓库没有引入单独的 ACL 模块，但 `interfaces/http` 实际上承担了一个**教学型的轻量 ACL 角色**：

- 把 HTTP 请求模型翻译为应用层命令 / 查询
- 把应用层结果投影为稳定的响应 JSON
- 把 FastAPI / Pydantic 的错误翻译为统一错误契约
- 把“空白字符串”等外部噪音吸收在边界层，不直接流入领域对象

因此这里并不是“路由顺手调一下用例”而已，而是在演示：

> 外部协议模型不应该直接穿透到内部领域。

## 内部模型边界

这个项目里至少存在 4 种不同目的的模型：

- **Request Model**：例如 `CreateTaskRequest`，服务于 HTTP 契约
- **Command / Query**：例如 `CreateTaskCommand`、`ListTasksQuery`，服务于用例输入
- **Domain Model**：例如 `Task`、`TaskTitle`，服务于业务规则
- **Response Model / View**：例如 `TaskView`、`TaskResponse`，服务于对外输出

它们看起来都在描述“任务”，但不应该合并成同一个类。

因为一旦合并，通常会立刻出现这些问题：

- 路由层和领域层互相知道太多细节
- 内部重构会意外破坏外部 API 契约
- 值对象与枚举被迫按 JSON 易用性牺牲语义表达
- 测试无法清楚区分“契约错误”与“业务错误”

## 为什么这个仓库会保持克制

目标不是做一个功能非常多的任务系统。

当前仓库里关于 CQRS / 领域事件的实现，也刻意保持在**教学版第二增量**：
- 领域事件在应用进程内同步分发
- projector 直接维护单表读模型
- 查询侧显式走 `TaskQueryService`
- 不实现事务 outbox、异步重试、跨进程消息投递

换句话说，它的价值在于把边界与职责讲清楚，而不是把生产级分布式一致性问题一次讲完。

目标是把这些东西讲清楚：
- 领域层该放什么
- 应用层该负责什么
- 基础设施如何通过端口接进来
- HTTP 层如何只做适配，不污染业务核心
- 上下文边界应如何通过 ACL 得到保护
- 错误码与状态码如何表达不同边界语义

因此这个仓库会优先保证：
- 结构可读
- 边界清楚
- 语义不被外部系统直接污染
- 容易扩展
- 适合教学和开源示例阅读
