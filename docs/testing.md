# 测试策略

这个仓库的测试目标不是“为了覆盖率而覆盖率”，而是保护示例项目最值得保留的东西：

- 业务规则
- 分层边界
- HTTP 契约
- 基础工程质量
- 新手可以顺利跑通的体验

## 当前测试分层

## 1. 领域层单元测试

关注纯业务规则，不依赖 HTTP、数据库或框架上下文。

适合覆盖：

- 标题不能为空
- 描述不能为空白字符串
- 指派人不能为空白字符串
- 已完成任务不能重复完成

这类测试通常最快，也最能直接保护业务核心。

## 2. 应用层用例测试

关注用例编排是否正确。

例如：

- 是否正确调用仓储
- 找不到任务时是否抛出预期异常
- 返回视图对象是否符合预期

这层测试用于保证“流程组织”没有偏离设计。

## 3. HTTP 接口集成测试

关注协议层契约。

例如：

- 状态码是否正确
- 成功响应是否统一包裹在 `data` 字段中
- 错误响应是否统一包裹在 `error` 字段中
- 输入校验失败时是否返回预期结构

## 4. 工程质量测试

这类测试不验证业务功能，而是保护仓库的工程约束。

例如：

- 文档是否存在
- 关键文件是否齐备
- README、Makefile、CI 是否使用一致口径
- 质量命令是否对新手足够直接

这类测试应该**克制**，重点是防止项目失去教学价值，而不是把示例仓库变成高门槛模板。

## 推荐执行方式

### 本地开发

至少执行：

```bash
make lint
make test
```

### 提交前

建议执行：

```bash
make check
make ci
```

其中 `make ci` 会执行与 CI 基本一致的检查流程：

```bash
python -m ruff check src tests
python -m ruff format --check src tests
python -m pytest --cov=task_management --cov-report=term-missing
```

## 质量门槛原则

### 为什么不强制单一覆盖率阈值

教学型仓库经常会发生下面几种情况：

- 新增了一段示意性代码，但还没到值得补足完整测试的阶段
- 文档和结构正在重构，短期内覆盖率会波动
- 某些适配层为了教学演示而存在，强行补测试收益不高

如果这时直接用固定覆盖率数字卡死 CI，往往会带来两个副作用：

1. 贡献者为了过门槛去写“只服务于数字”的测试
2. 新手误以为质量的核心是追数字，而不是守住关键行为

因此，本仓库默认保留覆盖率报告，但不把固定数字作为硬门槛。

### 真正应优先守住什么

- 关键业务规则不能失守
- 主要 API 主路径必须可运行
- 文档承诺的命令必须真实可用
- 分层边界不能被随意打穿

## 原则

- 优先保护业务规则
- 不让测试过度依赖基础设施细节
- 用最少的接口级测试守住关键主路径
- 当行为发生变化时，优先补测试再改实现
- 当质量规则与教学目标冲突时，优先保留教学可读性与可运行性

## 后续可补强方向

如果这个示例继续演进，建议逐步补充：

- 仓储契约测试
- 错误响应契约的更细粒度断言
- 更明确的测试数据构造工具
- 更清晰的本地开发排错指南

如果未来仓库定位发生变化，真的要向生产模板靠拢，再考虑逐步引入更严格的质量门槛，而不是在教学阶段一次性加满。
