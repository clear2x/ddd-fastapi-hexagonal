# 测试策略

## 测试金字塔

建议按下面 4 层推进：

1. 领域层单元测试
2. 应用层用例测试
3. API 集成测试
4. 最少量端到端冒烟测试

## 当前 MVP 重点

当前优先覆盖：
- 任务创建规则
- 任务完成规则
- 基础 API 生命周期
- API 契约层的错误语义与模型翻译

## 当前测试如何服务教学目标

这个仓库的测试不只是为了证明“接口能通”，还要帮助读者看见边界。

### 1. 领域测试：保护真正的业务不变量

领域层测试回答的是：

- 哪些值在业务上根本不成立
- 状态迁移是否被领域对象正确守住
- 不依赖 HTTP、数据库时，规则是否仍然成立

例如：

- 空标题不可创建任务
- 空白描述不能直接进入领域对象
- 已完成任务不能重复完成

### 2. 应用层测试：保护用例编排与输入输出形状

应用层测试回答的是：

- 用例是否通过命令 / 查询对象工作
- 用例是否正确调用仓储端口
- 用例返回的 View 是否适合作为对外投影的中间层

### 3. API 测试：保护对外契约，而不是保护框架细节

API 测试重点关注：

- 统一成功 / 错误响应结构
- `422 / 404 / 409 / 400` 的语义是否稳定
- 外部模型到内部命令的翻译是否正确
- 边界层是否吸收了空白字符串、缺字段等输入噪音

教学上尤其重要的一点是：

> API 测试不是为了证明 FastAPI 会工作，而是为了证明“接口边界翻译没有走样”。

## 下一步补强方向

后续会继续补：
- 仓储契约测试
- 更细粒度的错误响应契约测试
- CI 中的测试与静态检查

## 原则

- 优先保护业务规则
- 不让测试过度依赖基础设施细节
- 用最少的 E2E 测试守住关键主路径
