# 贡献说明

## 贡献目标

这个仓库的定位是“适合阅读、适合教学、适合继续演化”的最小示例项目。

因此在贡献时，优先考虑：

- 是否让边界更清楚
- 是否让接手者更容易理解
- 是否让运行和测试路径更稳定
- 是否让文档与代码保持一致

## 分支约定

如果你要发起一组相对完整的改动，建议基于最新 `main` 新建一个干净分支，例如：

- `feat/domain-core`
- `feat/http-api`
- `feat/tests-docs-ci`
- `feat/runtime-dx`
- `fix/api-validation`
- `chore/quality-guard-tune`

如果是临时修复，也可以使用更聚焦的命名，但应尽量一眼能看出改动方向。

## 协作规则

- Domain 层不要引入 FastAPI、SQLAlchemy 等框架依赖
- Application 层扩展后，尽量保持一个文件一个用例或一组强相关用例
- 任何行为变更，都应同步补测试或更新测试
- README 与 `docs/` 要和真实 API、真实结构保持一致
- **所有文档与代码注释统一使用中文**
- 类名、函数名、模块名、API 路径保持英文，便于开源协作与 Python 项目习惯对齐

## 提交前检查建议

### 先跑通最小检查

日常改动建议先执行：

```bash
make check
```

它适合快速确认：

- 静态检查是否通过
- 主要测试是否通过

### 提交前再跑 CI 对齐检查

准备提交或发 PR 前，建议执行：

```bash
make ci
```

该命令会尽量贴近 CI：

- `python -m ruff check src tests`
- `python -m ruff format --check src tests`
- `python -m pytest --cov=task_management --cov-report=term-missing`

## 关于覆盖率数字

这个仓库会保留覆盖率报告，方便观察测试是否明显退化；但它是一个教学示例，不建议把单一覆盖率数字当成唯一质量标准。

更重要的是：

- 关键教学路径是否仍被覆盖
- 文档宣称的行为是否真实可跑
- 新增约束是否会误伤第一次参与项目的人

如果你发现某条质量规则已经开始“为了卡门槛而卡门槛”，欢迎直接提出收敛建议。

## 提交 PR 前检查清单

- [ ] 已基于最新 `main` 同步分支
- [ ] 已运行 `make check`
- [ ] 如改动涉及 CI / 文档 / 质量规则，已运行 `make ci`
- [ ] 已检查 API 示例是否仍然正确
- [ ] 已检查架构边界是否仍然清晰
- [ ] 已检查新增文档与注释是否使用中文
- [ ] 已检查 README 与 `docs/` 是否需要同步更新

## 推荐贡献顺序

如果你是第一次参与这个仓库，建议优先从下面几类改动开始：

1. 文档补全与修正
2. 测试补充
3. 运行体验改进
4. 结构性重构
5. 新功能扩展

这样可以尽量避免在尚未理解边界时，过早把业务逻辑和框架细节耦合在一起。
